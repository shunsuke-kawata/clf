② JWT（JSON Web Token）の構造

JWTは . で区切られた3パートのBase64URL文字列。サーバーは署名を検証するだけで DB 照会不要。

③ Access Token / Refresh Token の使い分け
Access Token（clf_access） — 15分
15m
短命。盗まれても即座に使えなくなる。
Refresh Token（clf_refresh） — 30日
30d
長命。Access が切れた後にこれで再認証（再ログイン不要）。
// config.ts — APP_CONFIG.auth accessTokenExpiry: "15m", accessMaxAge: 60 * 15, // 900秒 refreshTokenExpiry: "30d", refreshMaxAge: 60 * 60 * 24 * 30, // 2592000秒
④ リクエスト時の認証検証フロー（verifySessionCookies）

getSession()（Server Component）/ getSessionRole(req)（API Route）から呼ばれる

🍪
Cookie から clf_access を取得
ブラウザが自動送信 → サーバーで読み取る
✅
Access Token の検証成功 → ロール返却
jwtVerify(token, secret) で署名・有効期限・sub を確認。OKなら即返却。
⚠️
Access Token が無効・期限切れ → Refresh Token にフォールバック
clf_refresh Cookie を読み、同様に jwtVerify で検証
✅
Refresh Token の検証成功 → ロール返却
30日以内であれば再認証なしでセッション継続
❌
両方無効 → null 返却（未認証）
呼び出し元がログインページへリダイレクト
// auth.ts — verifySessionCookies（実際のコード） async function verifySessionCookies(store: CookieGetter): Promise<SessionRole | null> { const accessToken = store.get(ACCESS_COOKIE)?.value; if (accessToken) { const role = await tryVerifyJwt(accessToken, ACCESS_COOKIE); if (role !== null) return role; // ← 成功：即返却 } const refreshToken = store.get(REFRESH_COOKIE)?.value; if (!refreshToken) return null; // ← Cookie 自体なし return tryVerifyJwt(refreshToken, REFRESH_COOKIE); // ← フォールバック }
⑤ ロール（admin / user）の違い
👑 admin ロール
管理者パスワード（ADMIN_PASSWORD）でログイン
ロッカーの投稿・編集・削除
/admin/* ルートへアクセス可
写真のアップロード・削除
👤 user ロール
一般パスワード（APP_PASSWORD）でログイン
ロッカーの閲覧のみ
地図の検索・操作
投稿・編集・削除は不可
🌐 未ログイン（null）
ロッカー一覧・詳細の閲覧（GET）のみ
ログイン不要でも地図は見える
書き込み系 API は 401 返却
⑥ ログアウト（destroySession）
// auth.ts — destroySession() export async function destroySession(): Promise<void> { const cookieStore = await cookies(); cookieStore.delete(ACCESS_COOKIE); // clf_access を削除 cookieStore.delete(REFRESH_COOKIE); // clf_refresh を削除 }

Cookie を削除するだけ。JWT はサーバー側にブラックリストを持たないのでトークン自体は期限まで有効だが、 Cookie がなければリクエスト時に届かないため実質無効になる。

⑦ セキュリティ上の工夫
🛡️
httpOnly Cookie — XSS 対策
JWT を httpOnly Cookie に保存することで、悪意ある JavaScript から document.cookie でトークンを盗まれるリスクをゼロにする。 localStorage に保存する実装より安全。
⏱️
timingSafeEqual — タイミング攻撃対策
通常の === 比較は一致しない文字で処理が止まる（最初の1文字で早期リターン）ため、 比較時間の差から正しいパスワードを推測できる。timingSafeEqual は常に全文字を比較し時間が一定。
🔄
Access + Refresh の2トークン — トークン盗難リスク低減
頻繁に使う Access Token を短命（15分）にすることで、万が一漏洩しても被害範囲を限定。 Refresh Token は長命（30日）だが、使用頻度が低くリスクが小さい。
🔏
sub クレームによるトークン流用防止
Access Token の sub は "clf_access"、 Refresh Token の sub は "clf_refresh" と異なる値。 tryVerifyJwt で payload.sub !== expectedSub のときは拒否するため、 Refresh Token を Access Token として使う流用攻撃を防ぐ。
